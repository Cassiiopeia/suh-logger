📝 현재 문제점
---

현재 suh-logger는 상위 스프링 프로젝트의 로깅 시스템과 **완전히 분리된 커스텀 로깅 인프라**로 동작한다.

- `java.util.logging`(JUL) 로거를 생성해 `System.out`에 **직접 출력**한다. (`SuhLoggerConfig.DirectConsoleHandler`)
- `SuhLogFormatter`가 ANSI 색상·타임스탬프·PID·스레드명·축약 클래스명을 **직접 생성**해 붙인다.
- `NoOpLoggingSystemFactory`를 스프링부트 `LoggingSystemFactory`로 등록하여, `getLoggingSystem()`에서 `null`을 반환해 **스프링부트의 Logback 초기화 자체를 막는다.**
- `build.gradle`이 `slf4j-api`, `logback-classic`, `spring-boot-starter-logging`을 **전부 exclude** 한다.

이로 인해 실사용에서 다음 문제가 발생한다.

- 상위 프로젝트에서 `@Slf4j` 로그를 외부(로그 수집기 등)로 전송하려 해도 **로그가 전송되지 않는다.** `NoOpLoggingSystemFactory`가 상위 프로젝트의 Logback 초기화를 막아, 상위가 설정한 패턴·레벨·Appender(외부 전송 포함)가 **모두 무력화**되기 때문이다. 즉 이 라이브러리가 상위 프로젝트의 정상적인 SLF4J 로깅까지 깨뜨린다.
- suh-logger 자체 출력이 상위 프로젝트의 로그 컨벤션(`logging.pattern.console`, `logback-spring.xml` 등)을 전혀 따르지 않아 포맷이 따로 논다.

이는 과거 이슈 #32(SL4J 의존성을 삭제하고 독립된 IntelliJ형 로그 형식으로 전환)의 방향을 되돌리는 **재전환** 작업이다. 당시 의존성 충돌 회피를 위해 독립화했으나, 상위 로깅까지 차단하는 부작용이 더 큰 문제로 확인되었다.

🛠️ 해결 방안 / 제안 기능
---

라이브러리의 로그 출력을 **상위 프로젝트의 SLF4J 파이프라인으로 위임**한다. 라이브러리는 메시지 텍스트만 생성하고, 색·타임스탬프·PID·클래스명·레벨 제어·Appender(외부 전송 포함)는 전부 상위 프로젝트가 담당한다.

- **위임 효과**: 전환 후 상위 프로젝트의 `logging.pattern.*`, `logging.level.kr.suhsaechan.suhlogger`, `logback-spring.xml`의 `<appender>`(파일·외부 전송·JSON 등)가 suh-logger 출력에 그대로 적용된다. 상위 `@Slf4j` 로그와 suh-logger 로그가 **동일 파이프라인·동일 패턴·동일 Appender**를 공유하게 된다.
- **하위호환 유지**: `SuhLogger`의 공개 API 시그니처를 100% 유지한다. 상위 프로젝트는 의존성 버전만 올리면 되고, 코드 수정은 필요 없다.

⚙️ 작업 내용
---

- `NoOpLoggingSystemFactory` 삭제 (상위 Logback 초기화를 막던 근본 원인)
- `SuhLoggerConfig` 삭제 (커스텀 JUL 로거 + ANSI 포맷터 + System.out 직접 출력)
- `META-INF/spring.factories`의 `LoggingSystemFactory` 등록 제거
- `build.gradle`의 로깅 exclude 블록 제거, `slf4j-api` compileOnly 추가
- `SuhLogger` 내부를 JUL → SLF4J로 전환 (고정 로거 이름 사용, 공개 API 시그니처 무변경)
- 내부 `{N}` 인덱스 플레이스홀더를 SLF4J `{}`로 정규화 치환 (기존 `{1}` 이상 인자 누락 버그 동시 수정)
- `setLogLevel` / `addFileLogger`는 `@Deprecated` no-op 처리 (시그니처만 유지, 상위 logging 설정으로 대체 안내)
- `SuhLoggerAutoConfiguration`의 SLF4J 차단 잔재(`System.setProperty` 등) 제거
- README 및 docs(api-reference, troubleshooting)의 JUL·독립 로깅 서술을 SLF4J 위임 기준으로 갱신
